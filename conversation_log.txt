Good project. Two key decisions to make before touching code:

---

## SQL Parsing

| Library | What it does | Verdict |
|---|---|---|
| **`sqlglot`** | Full AST parser, understands CTEs, joins, column aliases, multiple dialects (BigQuery, Snowflake, etc.) | Best choice |
| `sqlparse` | Tokenizer/formatter only — no real AST, can't reliably extract column semantics | Too weak |
| `antlr4` + grammar | Full parser generator, very powerful | Way too heavy |

**Pick `sqlglot`.** It gives you a proper AST so you can walk the tree and know which columns are selected, which come from joins, whether a CTE is referenced, etc. It also transpiles between dialects — useful if you're writing BigQuery SQL but running locally on DuckDB.

---

## Local SQL Execution

| Library | What it does | Verdict |
|---|---|---|
| **`duckdb`** | In-process OLAP engine, reads CSV/Parquet directly, excellent SQL coverage, fast | Best choice |
| `sqlite3` | stdlib, simple, but weak dialect support — missing window functions, array types, etc. | Too limited |
| `pandas` | You'd have to translate SQL → DataFrame ops | Not what you want |

**Pick `duckdb`.** It runs your actual SQL as-is (with minor dialect adjustments via sqlglot if needed), reads your sample CSVs natively with `read_csv_auto()`, and handles CTEs, window functions, and complex joins. You register your sample data files as tables and your query just runs.

---

## Interactive Approval

**`questionary`** — gives you a checkbox list out of the box (`questionary.checkbox(...)`). One import, looks great in the terminal. No need to build a custom "skip 3" parser unless you want to.

---

## Proposed project structure

```
query_tester_agent/
├── main.py                    # CLI entrypoint (argparse)
├── requirements.txt
├── sample_data/               # CSV files standing in for prod tables
│   ├── users.csv
│   └── orders.csv
├── example_queries/
│   └── sample.sql
└── src/
    ├── __init__.py
    ├── parser.py              # sqlglot: extract columns, joins, CTEs
    ├── suggester.py           # column name/type heuristics → test list
    ├── runner.py              # duckdb: execute query + run each test
    ├── interactive.py         # questionary: show tests, get approval
    └── reporter.py            # print pass/fail table (rich)
```

I'd add `rich` for the final report table — it's a natural pair with `questionary` and makes the output readable.
